# 01. コンセプト

## 解決したい問題

ソフトウェアの要件定義フェーズで、以下の問題が日常的に発生する。

1. **文字ばかりのドキュメントが頭に入らない**
2. **要件の粒度と書き方が統一されない**：Confluence に書き溜めていくうちに、ページごとに深さや書式がバラバラになる
3. **俯瞰して見るために図を描いてもしっくりこない**：draw.io や Figma で描いても「ただの絵」で終わってしまう
4. **課題管理と要件定義書が別々で、行き来が面倒**：Jira と Confluence のタブを往復しながら思考が分断される
5. **既存コードを読みながら新機能を設計する作業が非効率**：IDE と要件定義ツールを別に開き、頭の中で対応付ける必要がある

特に **既存システムへの機能追加** では、5番目が深刻になる。新規開発と違い、既存コードの振る舞いを前提に要件を組み立てる必要があるが、そのためのツールが存在しない。Confluence はコードを理解しないし、IDE は要件定義の場ではない。

## Tally が提供する体験

上記の断絶を埋めるため、Tally は以下を一枚のキャンバスに統合する。

- **要求** の本文
- **ユースケース** の構造
- **ユーザーストーリー** と受け入れ基準、実装タスク
- **論点**（まだ決めていない設計判断）とその選択肢
- **既存コード** の参照（ファイル/関数単位）
- **課題**（解決すべき問題）
- **AI提案**（Claude が生成した叩き台）

これらは型付きのノードとしてキャンバスに配置され、SysML 2.0 準拠のエッジで相互接続される。

### 典型的な使い方

1. 顧客や PdM から**要求書**（メール・Word・Markdown）が届く
2. Tally に投入すると、AI が要求を解釈して**論点カード**と**機能候補ノード**を配置
3. ユーザーは既存の Git リポジトリを紐付ける
4. AI が「この要求は既存のこのモジュールに近い」「ここは新規実装が必要」と**色分けして示す**
5. ユーザーはキャンバス上でカードを動かしたり繋いだりしながら要件を構造化していく
6. 曖昧な箇所にはコメントを残す → 自動的に**課題ノード**が生えて Jira に同期される
7. 未決定の設計判断は**論点ノード**として残り、選択肢を並べて後で決定できる
8. AI に「このユースケースを詳細化して」「このコードの責務を図に反映して」と図上で依頼する

### 差別化ポイント

既存ツールは以下の理由で、この体験を提供できない。

| ツール | できないこと |
|---|---|
| Figma / Miro / FigJam | 自由に絵が描けるが、意味を持ったノード構造を扱えない。コードを理解しない |
| Confluence | 文書に寄りすぎ、図は画像として埋め込むだけ |
| Jira | 課題は管理できるが、要求・UC・コードとの文脈が失われる |
| draw.io / Mermaid / PlantUML | 図は描けるが、コード連携・コメント・課題管理がない |
| Rational DOORS / Polarion | 重厚で日常開発には合わない、個人利用困難 |

Tally は **コードベースを第一級の入力として扱う要件定義ツール**という位置づけで、この隙間を埋める。

## 中核の思想

### 1. 思考のキャンバスそのものを成果物にする

従来の要件定義は「ドキュメント」を成果物とし、そこに至る思考プロセスは捨てられる。Tally ではキャンバス自体を成果物とし、**なぜその結論に至ったか**（どの論点でどの選択肢を選んだか、どの既存コードを参照したか）が残る。

### 2. AI が生成したものと人間が確定したものを明確に区別する

AI はあくまで叩き台を出す存在であり、意思決定者ではない。この区別を UI レベルで徹底するため、AI 生成ノードは必ず破線の **proposal** 型として表示し、人間が「採用」ボタンを押して初めて正規ノードになる。

### 3. 決定と未決定を一目で区別できる

要件定義の現場では「まだ決まっていないこと」が多数存在する。これを放置すると後工程で爆発する。Tally では **論点ノード（question）** を型として導入し、選択肢付きで未決定の設計判断を明示する。決定したら実線に切り替わり、残った選択肢は履歴として保持される。

### 4. キャンバスの認知負荷を保つ

全部をキャンバス上に出すと破綻する。以下の原則で情報階層を設計する。

- キャンバスには **要求 / UC / ストーリー / 論点 / コード / 課題 / AI提案** の7種のみ表示
- **受け入れ基準 / タスク / ストーリーポイント** はストーリーノードの内側の属性（詳細シートでのみ編集）
- **規格対応属性**（機能/非機能区別、品質カテゴリ、優先度）は型定義に持つが UI には出さない（将来実装）

### 5. Git 管理可能な単一ソース

プロジェクトは任意のディレクトリ（例: `~/.local/share/tally/projects/<slug>/`）に YAML ファイルとして保存される。プロジェクトとリポジトリは1対1で対応せず、複数のコードベースを参照できる。既知プロジェクト一覧はレジストリ（`$TALLY_HOME/registry.yaml`）で管理する。これにより、PRレビュー・CI連携・履歴追跡・差分確認がすべて可能になる。独自バイナリ形式は選ばない。

## スコープと非スコープ

### スコープに含む

- キャンバスUIとノード/エッジ編集
- Claude Agent SDK による AI 支援
- Git リポジトリの読み取り（tree-sitter, LSP 利用）
- YAML での永続化
- Markdown / Mermaid / Confluence 向け書き出し
- Jira へのストーリー同期（Phase 2）

### スコープに含まない

- プロジェクト管理機能（ガントチャート、リソース管理）
- アジャイルボード（Tally は Jira や Linear を置き換えない）
- リアルタイム協調編集（Phase 3 で検討）
- コード生成（Claude Code が別途担当する）
- テスト管理ツール（verify エッジで参照するのみ）
