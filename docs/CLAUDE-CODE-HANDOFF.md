# Claude Code への引き継ぎプロンプト

このリポジトリ（Tally）を Claude Code に実装させる際、最初のメッセージとして使ってください。コピー&ペーストで使える形式です。

---

## バージョンA: 標準（Phase 0 から着実に）

```
このリポジトリは Tally という要件定義ツールの設計書だけがある状態です。あなたにはこれを実装してもらいます。

まず以下を順に読んで、プロジェクトの全体像を把握してください：

1. README.md（プロジェクトの顔と名前の由来）
2. CLAUDE.md（あなた向けの指示書。最重要）
3. docs/01-concept.md（解決したい問題とツールの思想）
4. docs/02-domain-model.md（ノード型・エッジ型の意味論）
5. docs/03-architecture.md（システム構成）
6. docs/04-roadmap.md（実装の段階）
7. docs/adr/0001〜0003（重要な設計判断の背景）
8. docs/prototypes/tally-mobile.jsx（単一ファイルの動くプロトタイプ、設計意図の参考）

読み終わったら、以下のルールに従って作業してください：

- 必ず Phase 順に進める（Phase 0 → 1 → 2 → ...、飛ばさない）
- 各 Phase の完了条件をすべて満たしてから次に進む
- Phase 完了時に報告して、次に進む前に僕の確認を求める
- 技術選定や設計判断で迷ったら ADR を書いて確認を求める
- プロトタイプは設計意図の参考だが、本実装はパッケージ分割・型安全・テストを徹底する
- CLAUDE.md の「禁止事項」「コーディング規約」「コミット規約」を厳守

まずドキュメントを読み終わったら、以下を報告してください：
- プロジェクトの目的を1段落で要約
- Phase 0 で作るファイル一覧
- 着手前に確認したい点（あれば）

報告を受けて OK と返したら、Phase 0 の実装を開始してください。
```

## バージョンB: 慎重に（各ステップで都度確認）

Phase 1 以降を特に慎重に進めたい場合。

```
このリポジトリは Tally という要件定義ツールの設計書です。あなたに段階的に実装してもらいます。

【最初にやること】
1. README.md → CLAUDE.md → docs/ の順に全文読む
2. 読了後、以下を報告：
   - プロジェクトの目的と特徴（あなたの言葉で要約）
   - ノード型7種とエッジ型6種を表で整理
   - Phase 0〜8 のそれぞれで作るものを1行で列挙
   - 設計に違和感があれば指摘（遠慮なく）

【作業ルール】
- 報告を受けて僕が「OK」と返すまで、コードは一切書かない
- Phase は必ず順番通り、飛ばさない
- 各 Phase の開始前に「これから作るファイルと判断ポイント」を提示して確認を求める
- 各 Phase の完了時に「作ったもの・判断理由・確認事項・次のステップ」を報告
- 迷ったら質問、曖昧なら ADR

では docs を読み始めてください。
```

## 運用上のコツ

### 最初の報告を必ず求める

どのバージョンでも、**いきなりコードを書かせない**ようにしてください。Claude Code にドキュメントを要約させる工程を挟むことで、誤解や飛ばし読みが早い段階で発覚します。

### Phase ごとに会話を分けるのもあり

Phase 0 が終わったら、`/clear` で会話をリセットし、新しい会話で「Phase 1 を始めて」と指示する運用もおすすめです。コンテキストが長くなりすぎると Claude Code の判断精度が落ちるため、Phase 単位で会話を区切る方が品質が安定します。

### 進捗が想定より遅い・速い場合

- 遅い場合：Phase の粒度が大きすぎる可能性。該当 Phase のタスクを細分化する
- 速い場合：完了条件をきちんと満たしているか、テストを書いているか確認

### ADR を書かせる癖をつける

技術判断のたびに ADR を書かせると、後から読み返せる意思決定記録が溜まります。CLAUDE.md に「技術選定で迷ったら ADR」と書いてあるので、Claude Code は自然に書くはずですが、書いていなければ明示的に指示してください。

### ドッグフーディング

Phase 5 まで完成した時点で、**Tally 自身の次の機能追加**を Tally で要件定義してみるのが理想です。これが最大のテストになります。

## 困ったときの指示例

### 実装が迷走したとき

```
一旦止まってください。
直近の実装で書いた全ファイルと、各ファイルで何を目的としたかを報告してください。
僕が見てから次の指示を出します。
```

### 設計が間違っていることに気づいたとき

```
[間違いの内容] について、docs/ の関連箇所を更新してから実装を直してください。
まず更新案を提示、僕が承認してから実装に着手。
```

### Phase を跨いで戻りたいとき

```
Phase N に戻ります。
理由：[理由]
現在の実装のうち、Phase M 以降の部分は一旦そのまま残してください。
Phase N で作ったものを修正したら、影響のある後続 Phase のコードを直す。
```
